# InnerOS 林间世界｜下一阶段简版 PRD

> 版本：V0.1  
> 日期：2026-06-14  
> 阶段主题：不新增功能，先让已有世界可理解、可交互、可保存、可回访

## 1. 发心与阶段判断

InnerOS 的核心不是让用户管理更多信息，而是让真实记录逐渐形成有证据、可回看的自我理解。

林间世界的价值也不在“像游戏”，而在于把离散 Memo 变成一个可走近、可停留、可重新理解的个人世界。用户应当能感到：这里出现的内容来自我的生活，我能决定是否打开、如何解释、是否留下新的痕迹。

当前版本已经有合格的氛围入口、完整地图画面、移动、Memo 物件、篝火、池塘、共写、设置与降级页，但功能完成度与视觉完成度不一致。下一阶段不扩地图、不加玩法，目标是把现有承诺做实。

## 2. 本阶段目标

1. 统一地图画面、HUD、碰撞、区域触发和高亮圈的坐标。
2. 每个可见核心区域都至少具备明确反馈，不再让用户到达后无事发生。
3. 打通“靠近 -> 互动 -> 保存/放弃 -> 返回世界 -> 再次回访”的基础闭环。
4. 角色外观只保留两名已有完整素材的角色。
5. 游戏内 Memo 正确渲染 Markdown。
6. HUD 在所有地图正常态稳定显示，并明确异常态与沉浸弹层中的显示规则。

## 3. 范围约束

本期不做：

- 新地图、新区域、新角色、新物件类型；
- 在线联机、账号关系和远程双人；
- 等级、任务、资源、签到和数值成长；
- 环境音、天气、季节等新系统；
- 复杂建造、自由摆放和 AI 新能力。

允许删除、合并或暂时隐藏尚未闭环的入口，避免以“有按钮”代替“有能力”。

## 4. 现状问题与优先级

### P0：阻断核心体验

#### 4.1 地图空间映射不统一

当前区域 HUD、互动触发、碰撞与背景图地标由多套手写坐标分别维护。池塘最明显：

- HUD 的池塘区域中心与实际互动触发点不同；
- 湖边互动高亮圈被画到了工坊位置；
- 代码中的地图高度注释仍为 600，实际常量为 450，部分旧坐标已经超出当前世界。

要求：

- 建立一份唯一的区域空间配置，统一地标中心、HUD 范围、互动点、互动半径、碰撞体和高亮圈锚点；
- 基于当前 `twilight-world-map-v3.png` 重新标定六个区域；
- 高亮圈必须使用当前互动对象自己的锚点，不能使用二选一兜底。

验收：

- 玩家站在池塘、篝火、工坊视觉地标旁时，HUD 名称、互动提示和高亮圈同时指向同一地点；
- 离开区域后 HUD 在 300ms 内恢复“林间世界 / 独自漫游”；
- 桌面与移动端缩放后仍保持对齐。

#### 4.2 HUD 偶发不显示

当前 HUD 仅在 Canvas 正常分支渲染；加载遮罩层级高于 HUD，Canvas/API 失败则切到不含 HUD 的降级页。用户因此会把加载、异常或状态切换理解为 HUD 随机消失。

要求：

- 明确 HUD 状态：传送门不显示；世界加载完成后显示；全屏对话层可弱化但不随机消失；异常页提供等价的返回与设置入口；
- 世界数据、地图图片和 HUD 使用明确的 ready 状态，不以多个布尔值隐式组合；
- 加载超时后进入可恢复异常态，不能无限遮住 HUD；
- 检查移动端安全区、窄屏碰撞和层级。

验收：

- 连续进入/退出世界 20 次，无 HUD 空白；
- 模拟世界 API 失败、地图图片失败和慢加载时，都有明确状态和退出路径；
- HUD 不遮挡核心互动提示，弹层关闭后立即恢复。

#### 4.3 角色外观与实际素材不一致

配置和设置页展示 6 个角色，但 Canvas 只有两行真实角色素材，其他 4 个最终都会复用第二名角色的地图形象。

要求：

- 仅保留“漫游者、旅人”两名角色；
- 入口或首次进入时完成一次角色选择，之后可在设置中切换；
- 删除四个占位色块和相关伪差异；
- 角色选择只改变外观，不绑定人格、能力或对话风格。

验收：

- 所有入口、设置和地图中只出现两名角色；
- 头像、选择态和地图 Sprite 一致；
- 刷新和再次进入后保留上次选择。

### P1：补齐已有交互闭环

#### 4.4 游戏内 Memo 支持 Markdown

主站 Memo 详情已经使用统一 Markdown 渲染组件，游戏内 Memo 遭遇仍以纯文本 `<p>` 显示。

要求：

- 游戏内原始 Memo 复用现有 `MarkdownContent`；
- 支持标题、列表、引用、粗斜体、删除线、代码、链接和换行；
- 继承安全链接策略，不执行原始 HTML 和脚本；
- 羊皮纸容器补齐 Markdown 的字体、间距、列表和代码样式。

验收：

- 同一条 Memo 在主站详情与游戏内语义结构一致；
- 长文可滚动，表格/代码不撑破弹层；
- 空内容、HTML 导入内容和 Markdown 内容均可读。

#### 4.5 Memo 遭遇闭环

现状：可打开、补一句、带到篝火、隐藏；但“隐藏”只改当前内存，没有调用持久化接口，刷新后可能重新出现。

闭环：

`靠近物件 -> 阅读原文 -> 补充/带到篝火/隐藏/直接离开 -> 明确保存结果 -> 回到地图 -> 再次靠近验证`

要求：

- 隐藏、补充和授权均有明确成功/失败反馈；
- 隐藏必须持久化，并提供现有设置或记录入口中的恢复方式；若本期不做恢复，则暂时移除“隐藏”；
- 从 Memo 进入篝火时，自动携带当前 Memo，不让用户再次寻找勾选。

#### 4.6 篝火对话闭环

现状：四种对话姿态、Memo 授权和流式对话已存在；但未邀请同行者时标题仍写“同行者”，也可直接输入，模式语义不清。

闭环：

`坐到篝火边 -> 选择独处或邀请 AI -> 可选带入 Memo -> 对话/静坐 -> 主动结束 -> 回到原位置`

要求：

- “未邀请 AI”和“AI 同行中”使用不同空状态与文案；
- “只陪我坐着”不应出现强发送导向；
- 关闭面板时中止生成并保留已完成消息；
- 不新增总结产物，本期只保证进入、交流、停止、退出稳定。

#### 4.7 共写闭环

现状：可以选 Memo、双方写作、选择保留方式、生成物件；但同屏双人没有可达入口，默认独处时却以“另一个视角”代替第二人，产品身份不清。

本期决策：

- 不新增双人能力；
- 若不能在现有入口中稳定切换到同屏双人，则工坊仅保留“自己分两次写下两个视角”的明确模式，移除“身边的人”暗示；
- 已生成物件必须可再次靠近并看到来源 Memo 与共同文字；
- 保存失败时不得丢失当前草稿。

闭环：

`选择 Memo -> 写下两个视角 -> 确认保留方式 -> 选择已有物件 -> 保存 -> 地图出现物件 -> 再次打开核对`

#### 4.8 池塘闭环

现状：可写一段只存 `sessionStorage` 的文字，但用户无法再次查看、管理或理解它会保留多久。

本期决策二选一：

- A：明确为“一次性静坐便笺”，离开后不承诺回访，并在关闭前说明；
- B：接入现有世界对象/本地记录存储，使其可再次查看。

本期推荐 A，避免新增存储与管理功能。池塘的核心价值是允许不产出。

闭环：

`到池边 -> 静坐或写一句 -> 明确选择不保存/仅本次保留 -> 返回地图`

| 记忆花园 | Memo 物件可读，但区域本身无说明 | 保留物件交互；空花园提供明确空状态，不新增玩法 |

## 6. 统一交互规则

所有现有区域遵循同一套规则：

1. **可发现**：进入互动半径后，只出现一个明确提示。
2. **可进入**：键盘、点击和移动端按钮触发同一动作。
3. **可退出**：`Esc`、关闭按钮和遮罩行为一致，退出前处理未保存内容。
4. **可确认**：保存、放弃、失败都有明确反馈。
5. **可回访**：承诺持久化的内容刷新后仍存在；不持久化的内容提前说明。
6. **不误导**：无功能的视觉物件不出现高亮、手型或功能命名。

## 7. 建议实施顺序

### 第一批：稳定世界

- 统一坐标配置，修复湖边 HUD、触发点与高亮圈；
- 修复 HUD 生命周期、异常态和移动端层级；
- 角色缩减为两名并持久化选择；
- 游戏内 Memo 接入 Markdown。

### 第二批：补闭环

- Memo 隐藏/补充/带入篝火；
- 篝火邀请、停止、退出；
- 工坊身份与物件回访；
- 池塘一次性静坐语义。

### 第三批：删去假入口

- 清理不可达的双人状态；
- 隐藏或改名记忆林等未实现承诺；
- 清理无效角色配置、旧地图坐标和未使用组件。

## 8. 阶段完成标准

- 两名角色从选择到地图显示完全一致；
- 六个区域逐一走查，无错误 HUD、错误圈选或无解释的强互动暗示；
- Memo Markdown 在游戏内正确显示；
- Memo、篝火、工坊、池塘四条现有流程都能完整进入并安全退出；
- 所有“已保存”内容刷新后仍存在，所有“一次性”内容事前说明；
- 正常、慢加载、API 失败、图片失败、空 Memo、长 Memo、移动端七类状态均完成验收；
- 不增加新页面、新系统和新玩法。
