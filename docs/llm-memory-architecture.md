# InnerOS LLM 与记忆系统架构

## 目标

InnerOS 的核心不是“每次把更多笔记塞给模型”，而是持续维护一组有证据、可更新、可回溯的用户状态。

统一链路：

```text
用户记录
  -> 分析任务
  -> 单条事实提取
  -> 相关记忆召回
  -> 状态与假设更新
  -> 洞察/行动生成
  -> 后续记录自动验证
```

DeepSeek Chat Completions 是无状态接口。即使 `deepseek-v4-flash` 支持长上下文，长期记忆仍必须由 InnerOS 管理，不能依赖模型保存历史。

## 架构边界

### 1. LLM Gateway

唯一允许直接访问模型供应商的模块。

职责：

- 固定使用 `deepseek-v4-flash`，默认并强制非思考模式。
- 统一文本、JSON、流式调用。
- 统一超时、重试、错误归一化。
- 为每次调用记录任务类型、耗时、token 用量和结果状态。
- 后续承接并发限制、预算、缓存、Prompt 版本和模型切换。

业务分析器不得自行拼接 URL 或直接调用 `fetch`。

### 2. Memory Store

记忆不是一张无限增长的摘要表，分为四层：

| 层级 | 内容 | 更新方式 |
| --- | --- | --- |
| 原始记忆 | Memo、对话原文 | 只追加或显式编辑 |
| 情节记忆 | 某天发生的事件、决定、行动和结果 | 新记录后增量提取 |
| 语义记忆 | 目标、人物、项目、偏好、方法、长期主题 | 合并同类事实并保留证据 |
| 工作记忆 | 当前目标、阻力、待验证假设、近期行动 | 高频更新，可过期 |

所有推断型记忆都必须保存：

- `evidence_ids`
- `confidence`
- `first_seen_at`
- `last_confirmed_at`
- `status`
- `supersedes_id`
- `model_version`
- `prompt_version`

### 3. Analysis Jobs

自动分析通过持久化任务运行，不依赖某个页面保持打开。

首批任务：

| 任务 | 触发条件 | 产物 |
| --- | --- | --- |
| `memo.extract` | 新建或修改 Memo | 单条结构化结果 |
| `memory.link` | 单条提取完成 | 相关旧记忆及关系 |
| `memory.reconcile` | 出现新证据或反例 | 更新语义记忆 |
| `action.infer-outcome` | 新记录涉及未完成行动 | 执行与结果推断 |
| `digest.daily` | 每日或首页刷新 | 今日焦点 |
| `profile.consolidate` | 每周或累计一定新记录 | 用户模型新版本 |

任务必须支持：

- 幂等键
- 状态机：`pending/running/succeeded/failed/dead`
- 指数退避
- 最大重试次数
- 输入版本
- 任务依赖
- 调用成本统计

Next.js `after()` 只适合轻量、可丢失的请求后工作。正式的长期分析任务应由数据库队列和独立 worker 执行。

## 统一调用约定

业务模块调用 Gateway 时必须声明 `task`：

```ts
await completeJSON({
  task: 'memo.extract',
  messages,
  maxTokens: 1200,
});
```

任务名用于成本归因、失败观察和后续 Prompt 评测。禁止使用无语义的“general”覆盖正式业务调用。

Gateway 会在所有 DeepSeek 请求的首个 system message 中统一注入运行时上下文：

- 当前日期与星期。
- 当前时区，默认 `Asia/Shanghai`，可通过 `AI_TIME_ZONE` 覆盖。
- 简体中文地区基准。
- “今天、昨天、最近、本周”等相对时间的解释规则。
- 明确区分当前日期、笔记日期和外部世界事实。

只注入日期级信息，不注入每秒变化的当前时间，避免无必要的 Token 消耗和 Prompt 缓存失效。运行时上下文不能被当作用户经历或偏好证据。

## 建议数据模型

第二阶段新增：

```text
analysis_jobs
memory_items
memory_evidence
memory_relations
user_model_versions
actions
action_evidence
hypotheses
hypothesis_evidence
prompt_versions
```

不要立即把所有内容塞入一个 `user_profile` JSON。单体 JSON 难以局部更新、追踪证据和处理互相冲突的事实。

## 召回策略

一次分析的上下文由多个通道合并：

1. 关键词召回。
2. 向量语义召回。
3. 同主题、人物、项目召回。
4. 时间邻近召回。
5. 当前目标和工作记忆召回。
6. 反例召回。
7. LLM rerank 后控制最终上下文。

长上下文用于容纳更好的候选集，不用于跳过检索和记忆治理。

## 渐进实施

### Phase 1：统一入口

- LLM Gateway。
- 任务类型。
- 调用日志。
- 现有功能全部经由 Gateway。

### Phase 2：可靠任务

- `analysis_jobs` 数据库队列。
- 保存 Memo 后只入队。
- 单进程 worker、幂等与重试。

### Phase 3：长期记忆

- 事件、实体、目标、行动、假设模型。
- 新记录触发增量 reconcile。
- 洞察不再重复新增，而是更新版本和证据。

### Phase 4：自动验证

- 从后续记录推断行动结果。
- 自动寻找洞察反例。
- 周期性重建用户模型。
- 仅在高影响且证据冲突时询问用户。

## 当前明确不做

- 不把 LLM Tool Calls 当作任务调度器。
- 不在请求链中串行执行完整记忆重建。
- 不因为 1M 上下文而每次发送全部笔记。
- 不允许洞察覆盖原始事实。
- 不使用没有证据引用的长期用户结论。

## Memory Link v5

v5 在 v4 成本约束上补充：

- 新增 `state`，保存有原文证据的阶段性情绪、需要、阻力和处境。
- 情绪标签只用于定位，不能脱离原文直接生成记忆。
- 长笔记不再只截取开头，改为首尾段与高信号段组合，输入原文片段总量不超过约 4,200 字符。
- `event` 只保存决定、结果和转折，不保存重复体验与普通流水。
- 单条笔记不得新建 `pattern`；模式必须来自跨记录整合。
- 存量重建先保存影子提议，显式应用前完整快照记忆表，可整批回滚。
- v5.7 存量重建进一步将候选记忆压缩到 8 条、原文证据片段压缩到约 3,000 字符、输出上限压缩到 900 Token；已被 v5 应用过的笔记不会重复进入批次。

## Memory Link v4 成本约束

历史基线显示，`memory.link` 平均输入约 5,286 Token/次，主要来自过多候选记忆和重复证据。

v4 约束：

- 纯摘录、纯资料和无长期信号的极短记录由本地规则跳过。
- 新笔记原文最多输入 2,600 字符。
- 候选记忆从 30 条压缩到 12 条。
- 每条候选只携带 2 段、每段最多 160 字符的证据。
- 单次最多生成 4 条记忆，输出上限 1,400 Token。
- 新增 `belief/pattern/preference/constraint`，降低 `event` 类型过载。
- 没有任何原始证据的孤立记忆自动清理。

存量记忆不使用关键词规则直接批量改写语义。后续重整必须分批、可回滚，并记录新旧类型与合并依据。

## Conversation Retrieval v3

对话检索采用“本地多路召回 + DashScope 向量召回 + 条件式专用 Rerank”。

已实现：

- `rag.query` 输出结构化检索计划，区分事实、人物、枚举、时间线、变化、模式和行动查询。
- `text-embedding-v4` 为新笔记异步生成 1024 维向量；长笔记按 1200 字、180 字重叠切成最多 16 个语义块。分块原文、位置和向量写入 `memo_chunks`，检索后将真正命中的段落交给 Rerank 和回答模型。
- DashScope 单次 Embedding 最多提交 10 段，客户端自动分批；存量笔记通过 `memo.embed` 后台任务并发回填。
- `rag.query` 生成独立的 `resolved_query`，查询向量使用消解后的问题；LLM 未消解“这件事、他/她”等指代时，本地逻辑使用最近用户问题兜底。
- 关键词、长期记忆、向量三路候选在本地合并，并固定保留关键词候选进入重排，避免不同召回分数量纲互相挤压。事实与人物查询仍执行锚点门槛，低置信度语义相似不能直接充当事实证据。
- 检索词区分精确词、实体词和语义扩展词，泛词不能与明确实体同权。
- 候选不再按最新记录预截断，而是在全部命中记录上评分。
- 事实查询增加长短语锚点门槛，避免只命中一个弱相关短词。
- 没有有效结果时返回空上下文，不再自动补充近期笔记。
- 长笔记按命中位置截取原文窗口，不再固定只发送开头 800 字。
- 完全重复和标题摘要重复的记录在进入上下文前折叠。
- 枚举和事实查询验证主体、动作、对象关系；没有直接证据时返回空上下文，不允许只因主题相邻而制造引用。
- 删除针对具体姓名和恋爱记录的硬编码枚举规则，所有查询使用统一证据门槛。
- 变化和时间线查询在候选池中固定加入早期、居中和近期记录，最终证据覆盖不同阶段。
- “现在/当前”的行动问题额外召回近期行动项、待解决问题和仍有效的目标、项目、约束，并对陈旧记录设置时间门槛。
- `qwen3-vl-rerank` 只在枚举、变化、模式、时间线或本地排序接近时触发。
- 专用 Rerank 请求失败时回退到 DeepSeek `rag.rerank`，再次失败则保留本地排序。
- Rerank 最多输入 24 条短候选，只返回 ID；最终笔记上下文设 9,000 字符预算。
- `retrieval_runs` 保存查询计划、候选数、结果和命中分块，用于诊断召回遗漏。
- `npm run test:retrieval` 运行真实检索回归，覆盖无答案、行动转折、跨时间变化、当前目标、多轮指代和隐含语义。

Provider 配置统一收敛在 `src/lib/ai/retrieval-provider.ts`，密钥仅通过服务端环境变量读取。当前 SQLite 数据规模下采用进程内余弦相似度计算；当笔记规模超过约 1 万条后，再评估 sqlite-vec 或独立向量数据库，避免提前引入额外基础设施。

### 对话思考模式

- 对话默认继续使用 DeepSeek 非思考模式，控制首字延迟和 Token 消耗。
- 用户可在输入区按需开启“深度思考”。请求通过 `thinking: { type: "enabled" }` 发送，并使用 `reasoning_effort: "high"`。
- Gateway 将 `reasoning_content` 与最终 `content` 分成两类流事件；前端以可折叠的“思考过程”展示，不将推理文本混入最终回答。
- 推理过程保存在消息的 `reasoning_content` 字段中，历史对话可重新查看；旧消息通过数据库兼容迁移默认为空。
- 模型输出期间不再自动滚动到底部。用户发送消息时只进行一次初始定位，后续阅读位置由用户控制。
