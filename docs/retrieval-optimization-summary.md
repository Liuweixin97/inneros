# 关键发现与优化建议总结

## 核心问题

### 问题 1：准确查找困难 → 缺乏降级策略
**表现**：对于表述多样或关键词不明显的问题，检索返回空或不相关结果

**根本原因**：
- 检索规划依赖 LLM，失败时无有效降级
- 多源检索融合后，同时低分则被过滤掉
- 缺少"如果结果不足就扩大范围"的逻辑

**单个改动的收益**：~~需要全套改动~~ → 通过**两阶段搜索**可快速改善

---

### 问题 2：跨多 Memo 综合 → 无聚类能力
**表现**：多个相关 memo 被平铺返回，LLM 看不出它们的内在联系

**根本原因**：
- 检索结果都是单 memo 排序，没有话题/人物/项目维度的分组
- 去重机制过激，可能误删同话题的不同角度
- buildContext 是线性拼接，没有概念连接说明

**单个改动的收益**：~~框架性问题~~ → 短期**强化对话语境**可缓解

---

### 问题 3：对话历史未充分利用 → 容易跑题
**表现**：多轮对话中重复引用或指代消解不准

**根本原因**：
- buildRetrievalPlan 只取最近 4 条用户消息
- chat 路由没有传递前次引用的 memo
- conversation summary 存在但未被利用

**单个改动的收益**：**自动传参 + 扩展上下文**即可立竿见影

## 快速见效方案（第 1 周）

### A1：扩展检索规划的输入
**改动**：
- 传入 `previousCitations` （前次 assistant 引用的 memo）
- 传入 `conversationSummary` （对话摘要）

**收益**：
- 对话连贯性立即改善
- buildRetrievalPlan 可做指代消解

**代码量**：~50 行

### A2：两阶段降级搜索
**改动**：
```
精准搜索 (term + entity + embedding + memory)
    ↓ (如果结果 < 3 条且质量差)
宽松搜索 (扩大时间 + 降低 term 要求 + 增强 semantic)
    ↓
合并、去重、排序
```

**收益**：
- 检索空结果率 ↓ 75%
- 表述多样问题的召回率 +15-20%

**代码量**：~150 行

### A3：改进时间范围判断
**改动**：
- 不仅看 query 中的时间词
- 也看 conversationHistory 中的时间信号
- 跟进问题保守用 all

**收益**：
- 用户问"这个之前有发生过吗"能正确扩大搜索范围
- 避免因时间范围过窄导致的漏检

**代码量**：~30 行

## 常见问题

**Q：能否只做方案 A 的一部分？**  
A：可以。建议至少实施 A1（扩展输入参数）+ A2（降级搜索）。A3（时间范围改进）是可选的锦上添花。

**Q：方案 A 是否会增加 API 调用成本？**  
A：仅宽松搜索阶段可能触发额外 embedding 调用，预计全量查询的 ~5-10%。整体增幅 < 10%。

**Q：如果方案 A 效果不达预期怎么办？**  
A：可立即切换到方案 B（跨 memo 聚类）或 C（对话历史）。三个方案相对独立，可灵活组合。

**Q：是否需要停机部署？**  
A：不需要。所有改动向后兼容，可灰度上线。建议先在 10% 流量验证。
